## Problem Statement

1. **[Important]** Q: MPC wallet operations incur significant computational overhead from expensive cryptographic operations (zero-knowledge proofs, multi-round communication protocols, homomorphic encryption) translating to increased operational costs (server compute $5K-$50K/month for high-volume providers), higher transaction latency (2-15 seconds vs. <1s for EOA wallets), and throughput limitations (50-500 transactions/minute vs. thousands for traditional custody), impacting profitability for wallet providers, user experience for high-frequency operations, and scalability for mass adoption serving 100M+ users. Formulate a structured problem statement using the following [Input] fields.
   
   A:
   - **Brief description of the problem to be analyzed**: 
     MPC threshold signature schemes require computationally intensive cryptography fundamentally more expensive than single-signature operations [Blog: Cubist, 2024; Guide: Stackup, 2025]. Cost and performance impacts: (1) Cryptographic complexity where each signing operation involves multiple parties executing zero-knowledge proofs (Paillier encryption range proofs, commitment schemes), homomorphic encryption operations (multiplicative shares, additive masking), and multi-round network communication (4-9 rounds for CGGMP21/GG18 protocols) [Guide: Stackup, 2025], consuming 10-100× more CPU cycles than single ECDSA signature; (2) Server infrastructure costs where high-volume providers (Fireblocks processing thousands of transactions/hour, exchanges with MPC hot wallets) incur $5K-$50K/month in server compute costs (HSMs, specialized cryptographic acceleration hardware, high-memory instances for zero-knowledge proof generation) [Estimated from enterprise infrastructure costs]; (3) Throughput bottlenecks where secure implementations achieve 50-100 transactions/minute per MPC cluster vs. thousands/minute for traditional hot wallets [Guide: Stackup, 2025], limiting scalability for high-frequency use cases (trading operations, payment processing, DeFi liquidations); (4) Latency-security trade-off where providers face pressure to sacrifice security for performance (using "honest-but-curious" MPC achieving <3s latency vs. 10s for malicious-security protocols [Blog: Cubist, 2024]), optimization shortcuts (disabling expensive zero-knowledge proofs, reducing communication rounds, weakening cryptographic parameters), or network topology compromises (collocating shares for <2s latency vs. geographic distribution for resilience adding 5-10s [Guide: Stackup, 2025]); (5) Energy consumption where large-scale MPC operations (institutional custody processing 10K+ transactions/day) consume significant electricity for cryptographic computation, contributing to carbon footprint concerns and operational expenses; (6) Mobile device battery drain where MPC client operations (zero-knowledge proof generation, multi-round signing) on mobile devices consume 5-10× more battery than traditional signing, degrading user experience and limiting mobile adoption [Estimated from cryptographic workload]. Resolution requires protocol optimization (newer CGGMP21 4-round vs. GG18 9-round, cryptographic acceleration hardware, batching multiple signatures), infrastructure efficiency (cloud cost optimization, edge computing for reduced latency, hardware security modules with cryptographic co-processors), and realistic performance expectations (accepting 2-5s latency as acceptable trade-off for distributed security vs. insisting on <1s matching EOA wallets).

## Glossary

- **CGGMP21**: UC Non-Interactive, Proactive, Threshold ECDSA protocol requiring 4 communication rounds (more efficient than GG18's 9 rounds); state-of-the-art MPC protocol balancing security and performance.
- **Computational overhead**: Additional CPU time, memory, and network required for MPC threshold signatures (10-100× more expensive than single-signature operations) due to zero-knowledge proofs, homomorphic encryption, multi-round protocols.
- **GG18**: Threshold ECDSA protocol by Gennaro and Goldfeder (2018) requiring 9 communication rounds; widely deployed but slower than modern CGGMP21 protocol.
- **Honest-but-curious MPC**: Insecure MPC security model assuming parties follow protocol but may observe data; broken by single malicious deviation but achieves <3s latency vs. 10s for secure malicious-security protocols.
- **HSM (Hardware Security Module)**: Dedicated cryptographic hardware for secure key storage and operations; MPC-optimized HSMs with cryptographic co-processors cost $50K+ vs. $10K for standard models.
- **Paillier encryption**: Homomorphic encryption scheme enabling encrypted computation (multiplicative operations on ciphertexts); used in threshold ECDSA but 100-1000× slower than native field arithmetic.
- **Signature batching**: Amortizing fixed setup costs across multiple signatures in single ceremony (Schnorr signatures batch efficiently, ECDSA batching limited); achieves 20-40% efficiency gains.
- **Throughput**: Transactions per minute a system can process; MPC wallets limited to 50-500 txn/min vs. thousands/min for traditional custody due to cryptographic computation constraints.
- **Zero-knowledge proof**: Cryptographic proof enabling verification without revealing secrets; required in MPC for security but CPU-intensive (modular exponentiation over 2048-4096 bit integers consuming 100ms-1s per proof).
