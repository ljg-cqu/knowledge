# Protocol Fragmentation and Lack of Standardization in MPC Wallet Ecosystem

**Last Updated**: 2025-11-29  
**Status**: Draft  
**Owner**: Protocol Standards Team

## Problem Statement

1. **[Important]** Q: MPC wallet ecosystem suffers from protocol fragmentation with multiple incompatible threshold signature schemes (Lindell17, GG18/GG20, CGGMP21, FROST) lacking interoperability standards, preventing seamless wallet portability, creating vendor lock-in, increasing integration complexity for exchanges and protocols, and fragmenting liquidity across incompatible custody solutions serving est. 100M+ users and $50B+ assets. Formulate a structured problem statement using the following [Input] fields.
   
   A:
   - **Brief description of the problem to be analyzed**: 
     MPC wallets deploy diverse threshold signature protocols (Lindell17 2-of-2, GG18/GG20 n-of-n ECDSA, CGGMP21 threshold ECDSA, FROST threshold Schnorr) without standardized interfaces or interoperability frameworks, creating ecosystem fragmentation where users cannot migrate key shares between providers, exchanges must integrate separately with each vendor's API (15+ major providers × unique integration = massive engineering overhead), and institutional custody solutions become siloed. Critical impacts: (1) Vendor lock-in where users cannot switch providers without generating new keys and migrating assets on-chain (paying gas fees + disrupting established addresses) [Guide: Stackup, 2025]; (2) Integration burden where each exchange/protocol must support multiple incompatible MPC implementations (estimated 200-500 engineering hours per integration × 15 providers = 3,000-7,500 hours industry-wide) [inferred from integration complexity]; (3) Security inconsistency where different protocols offer varying security guarantees (GG18 vulnerable to key extraction [Paper: Fireblocks, 2023], CGGMP21 secure with proper implementation, older Lindell17 requires strict abort handling) without clear comparison framework for users; (4) Innovation barriers where new protocols (threshold EdDSA, post-quantum MPC) cannot achieve adoption due to lack of standardized migration paths. Resolution requires protocol-agnostic standards layer (similar to W3C WebAuthn for authentication), interoperable key share formats (encrypted portability between compliant implementations), and industry consortium defining minimum security requirements and testing frameworks.

## Glossary

- **CGGMP21**: UC Non-Interactive, Proactive, Threshold ECDSA protocol by Canetti, Gennaro, Goldfeder, Makriyannis, Peled (2021) with 4-round signing, enhanced security proofs; state-of-the-art MPC protocol.
- **FROST**: Flexible Round-Optimized Schnorr Threshold signatures; threshold signature scheme for Schnorr-based blockchains (non-ECDSA), offering efficient signing with minimal rounds.
- **GG18/GG20**: Threshold ECDSA protocols by Gennaro and Goldfeder (2018, 2020); widely deployed but vulnerable to key extraction attacks discovered 2023.
- **Key share portability**: Ability to migrate encrypted key shares from one MPC provider to another without generating new keys or moving assets on-chain; currently non-existent in MPC ecosystem.
- **Lindell17**: Two-party ECDSA protocol by Yehuda Lindell (2017) for 2-of-2 threshold signatures; requires strict abort handling to prevent key extraction via test messages.
- **Protocol fragmentation**: Proliferation of incompatible threshold signature implementations without standardized interfaces, creating vendor lock-in and integration complexity.
- **Threshold signature scheme (TSS)**: Cryptographic protocol enabling k-of-n parties to jointly sign transactions without reconstructing complete private key; foundation of MPC wallets.
- **Vendor lock-in**: Inability to switch MPC wallet providers without generating new keys and migrating assets on-chain (paying gas fees, losing address history).
- **W3C WebAuthn**: Web Authentication standard enabling interoperable cryptographic authentication across browsers, devices, and providers; model for MPC wallet standardization.
