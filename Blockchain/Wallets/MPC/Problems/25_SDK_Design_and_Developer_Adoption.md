# SDK Design and Developer Adoption for MPC Wallet-as-a-Service

**Status**: Draft

## Problem Statement

1. **[Important]** Q: MPC wallet providers expanding to SDK and Wallet-as-a-Service (WaaS) offerings for external developers (exchanges, DeFi protocols, startups) face critical challenges balancing developer integration speed (<1 week), security guarantees (multi-tenant isolation, audit requirements), and monetization models that incentivize adoption without driving cost-sensitive users to bypass security features. Formulate a structured problem statement using the following [Input] fields.
   
   A:
   - **Brief description of the problem to be analyzed**: 
     Organizations packaging internal MPC wallet infrastructure as SDK and WaaS offerings must enable rapid integration (<1 week for typical startup teams, <4 weeks for institutional clients with audit requirements) while maintaining strict multi-tenant security isolation and designing pricing models that achieve business goals (hundreds to thousands of active developers, several million dollars ARR within 1-2 years) without incentivizing clients to bypass security features to reduce costs. Current challenges include: (1) Integration friction: MPC complexity creates 2-4 week integration timelines, driving developer drop-off; (2) Security tier complexity: serving both $100K AUM startups and $100M+ institutional clients with single SDK creates security/cost trade-offs; (3) Pricing misalignment: per-signature pricing ($0.01-$0.50/signature) drives high-frequency users toward risky hot wallets to avoid costs; (4) Multi-tenant isolation: shared MPC infrastructure must prevent cross-tenant key leakage and side-channel attacks. This affects platform providers targeting developer ecosystems, DApp developers requiring embedded wallets, and institutional clients needing compliant custody solutions.
   
   - **Background and current situation**: 
     Wallet-as-a-Service (WaaS) enables developers to integrate MPC wallet functionality via APIs and SDKs without building infrastructure from scratch [Blog: Turnkey, 2024]. Benefits include developer focus on application logic rather than cryptographic complexity, reduced time-to-market (weeks vs months for in-house development), and professional security/compliance (SOC 2, regulatory certifications) [Blog: Para, 2025]. Major providers offer WaaS: Fireblocks, Coinbase WaaS, Turnkey, Cobo, MetaMask SDK, Krayon Digital [Multiple sources]. Typical WaaS architecture: client backend integrates SDK, provider manages MPC infrastructure (key generation, signing coordination, share storage), user devices hold one key share (2-of-3 or 3-of-5 configurations) [Guide: Cobo, 2024; Guide: Alchemy, 2025]. Developer integration challenges: (1) SDK complexity: MPC key management, threshold coordination, and blockchain transaction construction require 40-200 hours engineering effort [implied from integration timelines]; (2) Documentation gaps: many providers lack comprehensive guides for common use cases (embedded wallets, custodial vs non-custodial models, chain-specific integration) [Source: Supplementary]; (3) Testing environments: insufficient sandbox/testnet support for development and QA phases [implied from developer pain points]. Security tier requirements vary dramatically: startups need basic security (2-of-3 threshold, standard KYC), institutional clients require SOC 2 Type II audits, penetration testing, and custom compliance features [Blog: AWS, 2024]. Multi-tenant security challenges: shared MPC infrastructure must isolate tenants through cryptographic separation (unique key material per tenant), access control (API authentication, RBAC), and side-channel protection (timing attacks, resource contention) [Guide: Alchemy, 2025]. Pricing model tensions: per-signature pricing ($0.01-$0.50 observed range) creates predictable provider revenue but penalizes high-frequency use cases (gaming, micropayments), driving users toward cheaper hot wallets; AUM-based pricing (% of assets) aligns incentives for custody but difficult to enforce for non-custodial integrations; hybrid models (base subscription + usage tiers) balance predictability and scalability but increase pricing complexity [Source: Supplementary]. 2025 identified as breakthrough year for WaaS adoption, with mainstream DApps and financial institutions integrating embedded wallets [Blog: Para, 2025].
   
   - **Goals and success criteria**: 
     Integration speed: Typical startup team completes core integration in <1 week (min) / <3 days (target) / <1 day (ideal) from SDK installation to first testnet transaction (current: 2-4 weeks). Institutional integration: Enterprise clients with audit requirements complete integration in <4 weeks (min) / <2 weeks (target) / <1 week (ideal) including security review (current: 8-16 weeks). Developer ecosystem: Achieve ≥500 (min) / ≥1000 (target) / ≥2000 (ideal) active developers using SDK by Q4 2025 (current: most providers <100). Revenue targets: Reach $2M ARR (min) / $5M ARR (target) / $10M ARR (ideal) within 24 months of SDK launch. Security adoption: ≥80% (min) / ≥90% (target) / ≥95% (ideal) of integrated applications use recommended security features (threshold signatures, not bypassing to hot wallets) by Q4 2025. Pricing satisfaction: Developer NPS ≥30 (min) / ≥50 (target) / ≥70 (ideal) indicating pricing perceived as fair vs security value delivered. Multi-tenant isolation: Zero cross-tenant key leakage incidents, 100% pass rate on security audits for tenant separation by Q4 2025.
   
   - **Key constraints and resources**: 
     Timeline: Q1 2025 - Q4 2025 (9 months SDK development + documentation + developer relations); Budget: $500K-$1.2M for platform engineering (3-5 engineers), documentation and DevRel (2-3 technical writers + 1-2 developer advocates), sandbox infrastructure ($50K-$100K), security audits ($50K-$100K); Team: 3-5 platform engineers (SDK design, API development, multi-tenant infrastructure), 2-3 technical writers (documentation, tutorials, reference guides), 1-2 developer advocates (community engagement, integration support), 1 product manager (pricing strategy, tier definition), 1-2 security engineers (multi-tenant isolation, audit preparation); Technology stack: RESTful APIs or GraphQL for backend integration, SDK client libraries (JavaScript/TypeScript, Python, Go, Java), MPC infrastructure must support multi-tenant key isolation, sandbox/testnet environments for development, monitoring and analytics for usage tracking; Security requirements: Multi-tenant cryptographic isolation (unique key material per tenant), API authentication (OAuth 2.0, API keys with rotation), RBAC for team management, audit trails (immutable logs for SOC 2), penetration testing and third-party security audits; Pricing complexity: Must serve diverse segments ($10K-$100K AUM startups, $1M-$10M mid-market, $10M-$1B+ institutional) with different pricing sensitivities and compliance needs; Developer experience: Documentation must enable self-service integration (minimal support tickets), sandbox must offer unlimited free testing (no production key risk), error messages must be actionable (not cryptographic jargon); Backward compatibility: Pricing and API changes must avoid breaking existing integrations.

## Glossary

- **API (Application Programming Interface)**: Programmatic interface enabling developers to integrate WaaS functionality (key generation, signing, transaction broadcasting) into applications.
- **ARR (Annual Recurring Revenue)**: Total annualized revenue from subscriptions and recurring usage fees; key metric for SaaS business model viability.
- **AUM (Assets Under Management)**: Total value of digital assets secured by wallet; pricing basis for some custody providers (e.g., 0.1-1% of AUM annually).
- **DevRel (Developer Relations)**: Team responsible for developer community engagement, documentation, integration support, and advocacy for developer needs internally.
- **DX (Developer Experience)**: Ease of SDK integration, documentation quality, API clarity, error messages, testing tools; primary factor in developer adoption.
- **Multi-tenant architecture**: System design where single infrastructure instance serves multiple independent customers (tenants) with strict isolation (data, keys, access).
- **NPS (Net Promoter Score)**: Customer satisfaction metric ranging -100 to +100; scores >50 considered excellent, >70 world-class; calculated from "would you recommend?" survey.
- **RBAC (Role-Based Access Control)**: Authorization model assigning permissions based on user roles (admin, developer, auditor); required for team management in WaaS offerings.
- **Sandbox environment**: Testing infrastructure (testnet, mock blockchain) enabling developers to integrate and test without production key risk or real asset loss.
- **SDK (Software Development Kit)**: Client libraries, documentation, and tools enabling developers to integrate WaaS functionality in specific programming languages (JavaScript, Python, Go, Java).
- **Side-channel attack**: Cryptographic attack exploiting implementation details (timing, power consumption, resource contention) rather than mathematical weaknesses; risk in multi-tenant systems.
- **SOC 2 Type II**: Security certification for service organizations covering controls over ≥6 month period; institutional clients require reports for vendor approval.
- **Threshold signature**: MPC scheme requiring k parties out of n to cooperate for signing (2-of-3, 3-of-5); WaaS typically splits shares between user device, provider servers, and backup.
- **WaaS (Wallet-as-a-Service)**: Business model where providers offer MPC wallet infrastructure via APIs/SDKs, enabling developers to integrate without building from scratch.
