   - **Historical attempts and existing solutions (if any)**: Existing MPC consumer wallets have experimented with various techniques: social recovery mechanisms where user designates trusted guardians (friends, family, or professional guardians) to help with recovery (reduces single-device dependency but adds social coordination complexity, guardian availability and competence varies, observed recovery success rate 85-90% when guardians are engaged but 50-60% when guardians don't respond or are confused by process), cloud or email-backed shares where one key share automatically backed up to iCloud/Google Drive/Email (improves recovery success to 90-95% but introduces centralized attack vector, cloud account compromise becomes wallet compromise, tension with "non-custodial" messaging), biometric-gated recovery using device biometrics (fingerprint, FaceID) as authentication factor for recovery (improves user experience, leverages familiar UI patterns, but biometric security model varies across devices and has known attack vectors, estimated false accept rate 1:50,000 to 1:100,000 which is acceptable for individual device security but concerning when protecting significant funds), strongly guided onboarding wizards with progressive disclosure (step-by-step setup with explanations and visual aids, reduces cognitive load, has improved completion rates by 10-15% in A/B tests), and gamification or engagement techniques (achievement badges for completing setup, recovery drills with rewards, educational mini-games explaining key concepts, mixed success with 5-10% improvement in engagement but risk of seeming gimmicky). Some wallets have improved metrics through these approaches, but industry-wide data still shows 22% higher friction and dropout relative to traditional wallets with single seed phrases, and many users remain confused by recovery responsibilities and device dependencies. Internal A/B tests may have explored various optimizations: copy changes (simplifying language, using analogies like "your wallet is protected by multiple locks in different locations"), flow ordering (when to introduce different security options, whether to make certain features optional vs. mandatory), progressive onboarding (minimum setup first, additional security features later), visual design improvements (progress indicators, illustrations explaining distributed security), and automation (auto-detecting and proposing recovery options based on user's device and accounts). However, there is no single, proven pattern that simultaneously achieves target metrics for drop-off (<15%), recovery success (>95%), and perceived trust (8.0+/10), indicating fundamental tension between MPC's multi-party model and consumer expectations for simplicity.
