## Glossary

- **API rate limiting**: Restriction on request volume per time period imposed by RPC providers to manage load and prevent abuse, causing wallet access failures during high-traffic periods
- **Centralized RPC provider**: Third-party service (Infura, Alchemy) aggregating blockchain node access via APIs, creating single point of failure despite underlying blockchain decentralization
- **Cloud provider lock-in**: Dependency on specific cloud platform (AWS, Google Cloud) due to proprietary managed services, data transfer costs, and operational complexity of migration
- **Decentralized RPC**: Protocol-based node network (Pocket Network, Lava) using token incentives to distribute blockchain access, offering redundancy at higher cost/latency
- **Failover**: Automatic switching from failed primary infrastructure provider to backup provider, with cutover time determining user-perceived downtime (goal <5s)
- **Light client**: Blockchain access method downloading only essential data (headers, state proofs) instead of full chain history, reducing RPC dependency for read operations
- **Multi-provider redundancy**: Infrastructure strategy using ≥3 RPC providers with health monitoring and automatic failover, increasing costs 40-80% for availability improvement
- **Self-hosted node**: Wallet provider-operated blockchain full node, eliminating third-party RPC dependency at cost of $500-$2K/mo per chain plus DevOps overhead
- **Web3 infrastructure paradox**: Contradiction where decentralized blockchain protocols accessed through centralized infrastructure (cloud hosts, RPC aggregators), undermining decentralization principles
