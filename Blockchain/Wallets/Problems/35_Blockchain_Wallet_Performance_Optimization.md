## Glossary

- **Archive node**: Full node storing complete blockchain history including all historical states (900GB+ for Ethereum vs 200-300GB pruned)
- **Compact block filters (BIP 157/158)**: Privacy-improved light client protocol downloading client-side filters of all block addresses (~10GB initial sync)
- **Eclipse attack**: Attack on light client where malicious nodes provide false blockchain view, exploiting lack of full verification
- **EIP-1559**: Ethereum Improvement Proposal 1559 introducing base fee + priority fee model with algorithmic adjustment to congestion
- **Fraud proof**: Cryptographic proof that block contains invalid state transition, enables challenging incorrect blocks in optimistic systems
- **Layer 2 (L2)**: Secondary blockchain network built on top of main chain (Lightning for Bitcoin, rollups for Ethereum) for scalability
- **Mempool**: Memory pool of pending unconfirmed transactions waiting for block inclusion, analyzed for fee estimation
- **Merkle proof**: Cryptographic proof of transaction inclusion in block using hash tree, enables SPV light client verification
- **Pruned node**: Full node deleting old blockchain data after verification, retaining only recent state (50-100GB vs 500GB full)
- **Replace-By-Fee (RBF)**: Protocol allowing replacing pending transaction with higher-fee version to accelerate confirmation
- **Snapshot sync**: Fast synchronization starting from recent trusted state checkpoint rather than genesis block (2-6h vs 24-72h)
- **SPV (Simplified Payment Verification)**: Light client protocol verifying transaction inclusion via Merkle proofs without full blockchain (~80MB vs 500GB)
- **State expiry**: Proposed protocol change automatically deleting old unused state to bound storage growth
- **Stateless client**: Client verifying blocks without storing state using witnesses (verkle trees enabling 2-5GB vs 200-300GB storage)
- **Verkle tree**: Cryptographic commitment scheme enabling stateless clients with 100x smaller proofs than Merkle trees (~2-5GB storage)
