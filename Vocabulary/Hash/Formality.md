1. **Original Expression**: Hash collisions are really bad for security  
   **Context**: Discussing hash function vulnerabilities in technical conversation
   
   **Core Intent**: Express concern about the security implications of hash collisions
   
   **Casual**: Hash collisions are a huge security problem  
   *Characteristics*: Direct, simple vocabulary, emphatic adjective "huge"  
   *Use when*: Informal team discussions, casual tech conversations, peer communication
   
   **Semi-Formal**: Hash collisions present significant security concerns  
   *Characteristics*: Neutral tone, professional vocabulary, measured emphasis  
   *Use when*: Standard workplace communication, technical reports, professional emails
   
   **Formal**: Hash collisions constitute a critical vulnerability in cryptographic systems  
   *Characteristics*: Precise technical terminology, formal verb "constitute", authoritative tone  
   *Use when*: Academic papers, security audits, formal documentation, executive presentations
   
   **Humorous**: Hash collisions are like two strangers ending up with the same house key—awkward and dangerous!  
   *Characteristics*: Uses analogy, creates vivid imagery, maintains technical accuracy through metaphor  
   *Use when*: Presentations to non-technical audiences, teaching contexts, rapport-building  
   *Avoid when*: Security briefings, incident reports, compliance documentation
   
   **Lively**: Hash collisions can absolutely wreck your security—they're a nightmare scenario!  
   *Characteristics*: Emphatic language ("absolutely wreck"), dramatic emphasis ("nightmare scenario")  
   *Use when*: Warning colleagues, emphasizing urgency, engaging storytelling
   
   **Empathetic**: I understand your concern—hash collisions can indeed compromise security in serious ways  
   *Characteristics*: Acknowledges listener's perspective, validating language, gentle emphasis  
   *Use when*: Addressing security concerns, mentoring, discussing vulnerabilities with stakeholders
   
   **Register Selection Guide**:
   - *Power Distance*: Higher authority (executives, auditors) → Formal; Peers → Semi-Formal or Casual
   - *Social Distance*: External stakeholders → Formal; Team members → Semi-Formal; Close colleagues → Casual
   - *Gravity of Situation*: Critical security breach → Formal; General discussion → Semi-Formal; Brainstorming → Casual
   - *Cultural Context*: Academic/research settings → Formal; Corporate → Semi-Formal; Startup culture → Casual
   - *Medium*: Written security audit → Formal; Email to team → Semi-Formal; Slack message → Casual
   
   **Common Mistakes**:
   - Using casual language in security audits: "Hash collisions are bad" is too informal for compliance documentation
   - Over-formalizing peer communication: "Collisions constitute critical vulnerabilities" sounds pretentious in team chat
   - Using humor in crisis situations: Jokes about security flaws are inappropriate during active incident response
   
   **Cultural Notes**: North American tech culture tends toward casual; European and Asian corporate contexts often expect higher formality in security discussions
   
   **Context Examples**:
   - **Situation 1**: Writing security audit for banking client → **Appropriate Variant**: Formal ("constitute a critical vulnerability")
   - **Situation 2**: Team meeting discussing algorithm choice → **Appropriate Variant**: Semi-Formal ("present significant security concerns")
   - **Situation 3**: Chatting with coworker about code review → **Appropriate Variant**: Casual ("huge security problem")
