## Q1: Choosing core Fork senses for an ESL lesson

**EssenceDimensions**: [SignalVsNoise, ScopeBoundaries] | **Difficulty**: F | **RoleContext**: ESL teacher planning an A2 "Fork & choices" vocabulary lesson  
**Criticality**: [Blocks, Quantified] | **Stakeholders**: [Teacher, Students] | **EstimatedTime**: ~10–15 min

**Question (for candidate)**:  
You teach an A2 adult ESL class. The textbook unit on *fork* and choices lists more than 20 items: fork (noun, utensil), fork (verb), fork in the road, branch, split, junction, prong, tine, pitchfork, tuning fork, fork over, fork out, forked tongue, multi-pronged strategy, fork a repository, clone, branch (verb), divergence, and more.  
You only have one 40-minute lesson before a short writing task where students must describe a picture (road splitting, person choosing a career path, table set with fork and knife). Past classes said "too many words" and then used only *fork* for everything.  
Your coordinator asks you to keep just **8–10 truly essential items** for the core lesson and move the rest to later.

From this situation:
1. Identify the **3–5 most essential levers** that should shape which senses and words you keep.  
2. Group them into **2–3 non-overlapping clusters** and name each cluster.  
3. Explain **what you will consciously de-prioritize or ignore** in this first lesson, and why.

**Answer Key (~150–250 words)**:  
- **Target Essence Levers (3–5)**:  
  - Cover the literal utensil/tool sense (*fork*, *prong*, basic table phrase *knife and fork*).  
  - Introduce the spatial branching sense (*fork in the road*, *take the left/right fork*).  
  - Add one clear choice metaphor (life/career fork in the road).  
  - Optionally include one contrast word (*branch* or *split*) to avoid overusing *fork*.  
- **Clusters (2–3, MECE)**:  
  - *Tools & table*: fork (noun), prong/tine, knife and fork.  
  - *Paths & choices*: fork (verb), fork in the road, take the left/right fork, branch/split (one contrast word).  
  - Later cluster *Idioms & tech*: fork over/out, forked tongue, fork a repository.  
- **Decision Link**: These clusters decide which 8–10 cards go into the core lesson, so students can write things like "At the fork in the road he took the left path" without mixing money or deception idioms.  
- **Metrics & Priorities**: Success metric: in the picture-writing task, ≥80% of students correctly use at least one tool word and one branching word.  
- **Common Failure Modes**: Trying to teach all idioms at once; not drawing a clear line between literal and metaphorical uses.

## Q2: Clarifying "fork" vs "branch" in an open-source README

**EssenceDimensions**: [ClusterMECE, DecisionLevers] | **Difficulty**: I | **RoleContext**: Open-source maintainer revising CONTRIBUTING docs  
**Criticality**: [Blocks, Stakeholders, Action] | **Stakeholders**: [Maintainer, New contributor, Documentation writer, Product owner] | **EstimatedTime**: ~10–15 min

**Question (for candidate)**:  
Your project README currently says things like "Fork the repo and create a feature branch," "Clone this fork," and "Push changes upstream." New contributors on a bootcamp course keep asking whether they should fork, branch, or just clone. Different tutorials they read online use *fork*, *branch*, and *clone* with slightly different meanings.  
You have been asked to rewrite the contribution section so that a motivated beginner with basic Git knowledge can follow it without confusion.

From this situation:
1. Identify **3–5 essence levers** that should govern your use of *fork*, *branch*, *clone*, and related terms.  
2. Group them into **2–3 MECE clusters** (for example, "local copy vs remote", "shared branch vs independent fork") and name each cluster.  
3. Explain how each cluster **drives concrete wording decisions** in the README.

**Answer Key (~150–250 words)**:  
- **Target Essence Levers (3–5)**:  
  - Distinguish *clone* (make a local copy) from *fork* (create your own remote copy under your account).  
  - Reserve *branch* for temporary lines of work within a single repository.  
  - Use *upstream* / *origin* only where needed to describe where changes flow.  
- **Clusters (2–3, MECE)**:  
  - *Local vs remote*: clone (local working copy) vs fork (your own remote repo).  
  - *Shared vs independent work*: branch within the main repo vs long-lived fork when you do not control upstream.  
  - *Flow terms*: optional, carefully introduced for contributors who need to send pull requests back.  
- **Decision Link**: These clusters determine how you rewrite instructions: for example, "Clone this repository" if you expect branches, or "Fork this repository" if contributors do not have write access. You remove mixed phrases like "clone this fork" in favor of one main pattern.  
- **Metrics & Priorities**: Track "number of clarifying questions about fork/branch/clone" and "successful first pull requests" after the change.  
- **Common Failure Modes**: Treating *fork* and *branch* as loose synonyms; introducing every Git term instead of focusing on the few that change contributor behavior.

## Q3: Editing a speech that overuses "fork in the road"

**EssenceDimensions**: [SignalVsNoise, DecisionLevers] | **Difficulty**: I | **RoleContext**: Communications coach polishing an executive keynote  
**Criticality**: [Risk, Stakeholders, Action] | **Stakeholders**: [Executive, Communications team, Audience] | **EstimatedTime**: ~10–15 min

**Question (for candidate)**:  
An executives draft keynote about company strategy uses many path metaphors: *fork in the road*, *crossroads*, *turning point*, *new path forward*, *no turning back*, and *the road not taken*. In several paragraphs, "fork in the road" appears two or three times, sometimes next to *crossroads* in the same sentence. Listeners in rehearsal say it "sounds dramatic but muddy" and they are not sure what exactly the choices are.

From this situation:
1. Identify **3–5 essence levers** that should shape how you use *fork in the road* and related metaphors in the final speech.  
2. Group them into **2–3 non-overlapping clusters** and name each cluster.  
3. Explain **which metaphors you will keep, which you will drop**, and how that clarifies the actual decisions being described.

**Answer Key (~150–250 words)**:  
- **Target Essence Levers (3–5)**:  
  - Use one primary path metaphor ("fork in the road" or "crossroads") consistently to signal a small number of concrete choices.  
  - Tie the chosen metaphor to explicit options (for example, "stay a niche tool" vs "become a platform").  
  - Avoid stacking similar metaphors that do not add new structure.  
- **Clusters (2–3, MECE)**:  
  - *Core image*: pick one main term (fork in the road or crossroads) and define what the branches actually are.  
  - *Supporting language*: keep a few neutral words (choice, direction, path) that echo the main image without competing.  
  - *Cut list*: drop extra metaphors ("road not taken", "no turning back") unless they mark truly different ideas.  
- **Decision Link**: These clusters determine specific edits: you rewrite paragraphs so each "fork in the road" sentence names the two or three real options, not just the feeling of drama.  
- **Metrics & Priorities**: Success metric: after rehearsal, listeners can state the main choice in one sentence.  
- **Common Failure Modes**: Treating metaphors as decoration instead of structure; using several near-synonyms that blur, rather than sharpen, decisions.

## Q4: Deciding whether to teach "fork over/out" in Business English

**EssenceDimensions**: [MetricsPriorities, ScopeBoundaries] | **Difficulty**: I | **RoleContext**: Business English course designer  
**Criticality**: [Risk, Stakeholders, Quantified] | **Stakeholders**: [Course designer, Teachers, Learners] | **EstimatedTime**: ~10–15 min

**Question (for candidate)**:  
You are designing a B1–B2 Business English unit on money verbs for email and meetings. Draft lists from colleagues include *pay*, *spend*, *disburse*, *remit*, *settle an invoice*, *fork over*, *fork out*, *cough up*, and *shell out*.  
The client company wants learners to write polite emails to customers and internal partners, but also understand informal speech from colleagues and podcasts. You have room for about 10–12 target expressions in this unit, including examples and practice time.

From this situation:
1. Identify **3–5 essence levers** that should decide whether *fork over/out* belongs in this unit.  
2. Group them into **2–3 MECE clusters** and name each cluster.  
3. Propose **simple metrics and priorities** for what goes into the core syllabus vs optional notes.

**Answer Key (~150–250 words)**:  
- **Target Essence Levers (3–5)**:  
  - Register fit: neutral/formal verbs for email vs informal idioms for listening.  
  - Frequency and usefulness in real business contexts.  
  - Risk of learners overusing rude-sounding phrases in writing.  
- **Clusters (2–3, MECE)**:  
  - *Core polite verbs*: pay, settle an invoice, remit, disburse – used in emails and documents.  
  - *Spoken-color verbs*: shell out, cough up, possibly fork out/over, taught explicitly as informal for meetings/podcasts only.  
  - *Out-of-scope for this unit*: very slangy or culture-specific phrases that distract from polite communication goals.  
- **Decision Link**: These clusters decide that *fork over/out* may appear as low-priority listening items with strong warnings about tone, or be left for a later "colloquial money talk" unit, while core time goes to neutral verbs that learners need every day.  
- **Metrics & Priorities**: Measure success by email samples: ≤5% of learner emails contain inappropriate money idioms; ≥80% correctly use at least two neutral payment verbs.  
- **Common Failure Modes**: Treating all synonyms as equal; giving fun idioms so much attention that learners choose them in formal mail.

## Q5: Standardizing "fork" language in an architecture decision record

**EssenceDimensions**: [ClusterMECE, MetricsPriorities] | **Difficulty**: A | **RoleContext**: Senior engineer writing an Architecture Decision Record (ADR)  
**Criticality**: [Blocks, Stakeholders, Quantified] | **Stakeholders**: [Architect, Engineers, PM, Ops] | **EstimatedTime**: ~10–15 min

**Question (for candidate)**:  
You are documenting a major architectural decision. Drafts of the ADR describe options using mixed vocabulary: *fork the service*, *spin off a new variant*, *branch the codebase*, *split the deployment*, and *take a multi-pronged approach*. New team members ask whether "forking" here means a permanent open-source fork, a feature branch, or a separate deployment per customer.

From this situation:
1. Identify **3–5 essence levers** that should govern your use of *fork*, *branch*, *split*, and *multi-pronged* in the ADR.  
2. Group them into **2–3 non-overlapping clusters** and name each cluster.  
3. Propose **1–3 simple metrics** to check whether the document's vocabulary is clear and consistent.

**Answer Key (~150–250 words)**:  
- **Target Essence Levers (3–5)**:  
  - Permanence of divergence (temporary branch vs long-lived fork vs just a configuration variant).  
  - Scope of the fork (code repository, runtime deployment, or organizational ownership).  
  - Audience familiarity with Git vs general systems language.  
- **Clusters (2–3, MECE)**:  
  - *Version-control terms*: branch, fork (repository-level only, if truly independent), with definitions up front.  
  - *Runtime topology*: duplicate vs shared services, using neutral words like "separate instance" instead of "fork the service" where possible.  
  - *Strategy metaphor*: keep or drop "multi-pronged approach" depending on whether it adds structure (distinct workstreams) or just color.  
- **Decision Link**: These clusters guide edits so "fork" is used only for repository-level independence, "branch" only for temporary code lines, and deployment changes are described with plain terms. This reduces misreads when people act on the ADR.  
- **Metrics & Priorities**: Metrics include "number of distinct terms for the same concept" (target ≤2) and a quick comprehension check: new engineers can correctly paraphrase what "fork" means here.  
- **Common Failure Modes**: Letting colorful metaphors leak into precise technical sections; reusing *fork* for any kind of split, making operational decisions harder.
